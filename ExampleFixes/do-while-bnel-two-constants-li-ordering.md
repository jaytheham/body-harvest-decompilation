### `do { bnel; bnel } while (ctr--)` loop: two pre-loop li ordering stubborn case

When a `do { } while (ctr--)` loop uses two pre-loop constants — a comparison constant (e.g. `tyCmpVal = 1`) and the loop counter (`ctr = 0x7F`) — the IDO compiler may emit `li ctr, 0x7F` **before** `li tyCmpVal, 1` even though the comparison value is initialized first in C source order.

**Pattern:**
```asm
addiu v0, v0, -offset  ; finish pointer load
li    a0, 1            ; TARGET: comparison constant first  ← wants this order
li    a1, 0x7f         ; TARGET: loop counter second
loop:
  lbu t7, 1(v0)
  move v1, a1          ; dead copy of counter (do-while artifact)
  bnel a2, t7, end     ; check entry id
  addiu v0, v0, 6
  lbu t8, 0(v0)
  bnel a0, t8, end     ; check type == 1 (uses a0)
  ...
end:
  bnez a1, loop        ; counter check (uses a1)
  addiu a1, a1, -1
```

**Root cause:** IDO treats the `move v1, a1` dead-copy (generated by the do-while lowering) as a first "use" of `a1`, making it appear earlier than `a0`'s first use (`bnel a0, t8`). However, in the target binary the `move v1, a1` artifact is NOT treated as a first-use for initialization scheduling — IDO instead considers the first non-dead use of `a1` (at `bnez a1`) which comes *after* `a0`'s first real use (`bnel a0, t8`), giving `a0` priority.

**Status:** Stubborn — after 10+ attempts varying:
- variable name (register naming trick: `a0`, `a1`), 
- declaration order, 
- initialization order, 
- pointer type (`MissionCondEntry*` vs `u8 *`), 
- comma-separated initializations, 
- function parameter types (`u8`, `s32` first arg),
- comparison operand order (`v0[0] == a0` vs `a0 == v0[0]`),
- loop variable type (`s32` vs `u8`)

**Fix:** Wrap in `#ifdef NON_MATCHING`. The C code is semantically correct; only the two pre-loop `li` instructions are in the wrong emission order.
