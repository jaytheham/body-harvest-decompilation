### Switch statement jump table blocked by binary blob placement

When a function contains a `switch` statement and the target jump table (`jtbl_XXXXXXXX_YYYYYYY`) is
at an address inside the `2EA10.bin` binary asset (RAM range 0x8002DE10–0x8003FB20), the C compiler
generates a new jump table that goes into `.rodata` at a different (later) address (0x8003FB20+).

**Symptom:**
- `lui at, 0x8003` in target vs `lui at, 0x8004` in current
- `lw t7, 0x6c8c(at)` in target vs different offset in current
- ROM checksum fails

**Root cause:** The address `jtbl_XXXXXXXX_YYYYYYY` is inside the `assets/2EA10.bin` binary blob.
The C compiler places its auto-generated switch jump table in `.rodata` after the blob ends.

**Resolution (fully matched):** Split `assets/2EA10.bin` around the jump table:
1. Compute the split offset: `splitOffset = jtbl_RAM_addr - 0x8002DE10` (blob RAM start).
2. Create `assets/2EA10a.bin` (first `splitOffset` bytes) and `assets/2EA10b.bin` (bytes after the
   `0x60`-byte jump table).
3. Update `bh.ld` to replace the single `build/assets/2EA10.bin.o(.data)` entry with:
   ```
   build/assets/2EA10a.bin.o(.data);
   build/src.us/core/53F0.c.o(.rodata);   ← inlined in data section
   build/assets/2EA10b.bin.o(.data);
   ```
   and remove `build/src.us/core/53F0.c.o(.rodata);` from the `_coreSegmentRoDataStart` block.

The jump table VALUES generated by IDO are byte-identical to the original blob data because the
function code has the same RAM addresses for each case label. Confirmed with `func_80007728_8328`
(`jtbl_80037140_37D40` at ROM 0x37D40, split offset `0x80037140 - 0x8002DE10 = 0x9330`).

**Companion fix — `addu` register order for `D_8004D160[idx]` lookup:**
After fixing the jump table, the only remaining diffs were `r` (register order) in the `addu`
for case 0x9B: `addu t2,v1,t1` (target) vs `addu t2,t1,v1` (current). The cause: D_8004D160 was
declared `extern u8 D_8004D160;` (scalar). Using `(&D_8004D160)[idx*2+1]` or pointer arithmetic
forms puts the freshly-computed sll result as `rs` (first). Fix: change declaration to
`extern u8 D_8004D160[];` and access as `D_8004D160[arg0[1] * 2 + 1]` (array subscript form).
Array subscript naturally places the array base (v1) as `rs` in the `addu`, matching target.

**Example:** `func_80007728_8328` in `src.us/core/53F0.c` (jump table `jtbl_80037140_37D40`)
**Also:** `func_80002EF8_3AF8` in `src.us/core/1050.c` (blocked, not yet split)
