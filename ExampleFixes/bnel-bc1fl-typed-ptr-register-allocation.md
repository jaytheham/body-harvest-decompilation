### `bnel` + `bc1fl` compound float/int condition: register allocation stubborn case

When a function uses a **compound OR condition** mixing int-equality and float comparisons (pattern: `if ((A==B && C==D) || ((temp = D, -1.0f == temp) && B < A) || (-1.0f == temp && B == -1))`):

- The `bc1fl` pattern is generated by `(float_assign_comma_expr && slt_cond)` — the `&&` after the comma float assignment produces `bc1fl`.
- The **m2c comma expression form** `(temp_f0 = ptr->unk10, (temp_f0 == -1.0f))` is essential; restructuring as nested ifs loses `bc1fl` and instead yields `bc1f`.
- Yoda condition `(-1.0f == temp_f0)` vs `(temp_f0 == -1.0f)` controls `c.eq.s` operand order (`c.eq.s $f2,$f0` vs `c.eq.s $f0,$f2`) but may not affect score if IDO optimizes both the same way.

**Stubborn case: `lw a2,...` vs `lw t8,...` register for the `bnel` comparison:**

- IDO may allocate register `a2` (argument register $6) for a field load from `arg1` when a `Unk8013E45C *temp_v0` pointer variable is declared—even when `arg1` is typed as `Unk80157600 *` with named field `unk298`.
- The target uses `t8` ($24) for the same load, due to the "next fresh T register after t7" allocation strategy.
- Removing the `temp_v0` pointer variable (using `D_arr[i].field` style) gives a worse score.
- The consequence of `a2` vs `t8`: the compiler deduplicates the two `arg1->unk298` accesses (first for `bnel`, second for `slt`) into one register, placing `slt` BEFORE `c.eq.s` instead of inside the `bc1fl` body. Target loads `t9` for the second access inside `bc1fl`.
- Resolution: **NON_MATCHING** if register allocation cannot be forced to t8.

**Key observation**: the `||` expression is critical for generating a single `bc1fl`. Using nested `if` blocks produces `bc1f` (not likely) without the `slt` reload inside the block.
